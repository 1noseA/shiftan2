---
name: create-issue
description: |
  GitHubにIssueを作成するスキル。会話の内容からIssue本文（日本語）を自動生成し、
  ユーザーに内容を確認させてから `gh issue create` を実行する。
  以下のような発言でこのスキルを使う：
  "Issueを作って"、"Issue作成"、"Issue出して"、"GitHubにIssueを立てたい"、
  "タスクをIssueにして"、"課題を登録して"、"チケットを作って"
  スプリント番号を引数で渡す場合（例：`/create-issue Sprint 2`）は
  docs/tasks.md から該当スプリントのタスク一覧を読み込んでIssueを生成する。
---

## 手順

### 1. コンテキスト収集

**引数にスプリント番号が指定された場合（例：`Sprint 2`）：**

`docs/tasks.md` を読み込み、該当スプリントのタスク一覧を取得する。
タスク一覧を「やること」の元ネタとして使い、ドラフトを自動生成する。
ユーザーへの追加確認は不要。

**引数がない場合：**

会話の流れからIssueの内容を把握する。不明点があれば以下を確認する：

- **タイトル**：何をするIssueか（例：「Sprint 2：シフト条件設定機能の実装」）
- **背景・目的**：なぜこのIssueが必要か
- **やること**：具体的な実装・対応内容
- **やらないこと**：スコープ外であることを明示したいもの
- **受け入れ条件**：完了の定義（テスト可能な形で書く）
- **補足**：仕様メモ・参考資料・懸念点など（なければ省略）

会話からある程度読み取れる場合は確認せずドラフトを作成してよい。

### 2. Issueドラフト作成

収集した情報をもとに **日本語** で下書きする。

**タイトル（70文字以内）**
スプリント名・機能名など「何をするか」を端的に。
例：`Sprint 2：シフト条件設定機能の実装`、`fix：シフト生成APIの認証エラーを修正`

**本文テンプレート：**

```
## 背景・目的
（なぜこのIssueが必要か。解決したい課題や達成したいゴールを書く）

## やること
- （実装・対応する内容を箇条書き）

## やらないこと
- （スコープ外を明示。「今回は〜はやらない」と書く。ない場合は「特になし」）

## 受け入れ条件
- （完了の定義をテスト可能な形で書く。例：「〜ができること」「〜が表示されること」）

## 補足
（仕様メモ、参考資料、懸念点などがあれば書く。なければこのセクションごと省略）
```

### 3. ユーザーへの確認

ドラフトを以下の形式で提示し、承認・修正・キャンセルを求める。
`gh issue create` はこの確認が取れるまで **絶対に実行しない**。

---
**【Issue確認】**

**タイトル：**
```
<タイトル>
```

**本文：**
```
<本文>
```

このIssueを作成しますか？（「はい」で作成 / 内容を指示すれば修正 / 「キャンセル」で中止）

---

修正を求められたら内容を反映して再度提示する。承認されたら次のステップへ。

### 4. Issue作成

```bash
gh issue create --title "<タイトル>" --body "$(cat <<'EOF'
<本文>
EOF
)"
```

作成後、返ってきた **Issue の URL** をユーザーに伝えて終了。

## エラー対応

| 状況 | 対応 |
|---|---|
| `gh: command not found` | `brew install gh` を案内 |
| 未認証エラー | `gh auth login` を案内 |
| リモートリポジトリ未設定 | `git remote -v` で確認し、GitHubリポジトリのURLを設定するよう案内 |
